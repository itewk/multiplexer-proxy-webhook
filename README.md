# multiplexer-proxy-webhook
If used as the webhook target of config.openshift.io/v1/Authentication then this webhook proxy
allows for authenticating both to the OpenShift Integrated Oauth, such as for UI access with any
OpenShift supported Identity Providers, and also configure a custom authentication webhook for
JWT based access to the OpenShift/Kube CLI.

## Usage Warning
This is a demonstration of capability and has not been vetted for production use.

## Building
Instructions for building this repository for development and deployment purposes.

### Binary
Build the go binary
```sh
go mod tidy
go build .
```

### Container
Building and pushing to OpenShift default container registry
```sh
export REGISTRY_URL=$(oc get route default-route -n openshift-image-registry --template='{{ .spec.host }}')
export MULTIPLEXER_PROXY_WEBHOOK_IMAGE_VERSION="latest"
export MULTIPLEXER_PROXY_WEBHOOK_IMAGE_PATH="multiplexer-proxy-webhook/multiplexer-proxy-webhook"

go mod tidy
podman build . -t ${REGISTRY_URL}/${MULTIPLEXER_PROXY_WEBHOOK_IMAGE_PATH}$:${MULTIPLEXER_PROXY_WEBHOOK_IMAGE_VERSION}
podman push ${REGISTRY_URL}/${MULTIPLEXER_PROXY_WEBHOOK_IMAGE_PATH}$:${MULTIPLEXER_PROXY_WEBHOOK_IMAGE_VERSION}
```

There is no reason this image has to be pushed to internal registry, it could be pushed to an external one,
but, the deploy resources currently have path hard coded for internal registry so that would need to be updated.

## Deploying

### Warm Deployment
This will deploy the multiplexer-proxy-webhook in a "Warm" way, that is you can test authenticating to it, but have not actually configured OpenShift Authentication to use it yet.

A [deploy](./deploy/) set of kustomize resources is provided, they are currently hard coded to my test environment
with all sorts of hard coded URLs and such. Maybe one day i'll get around to turning into a templated helm chart
but today is not that day. PRs welcome.

So if you want to deploy you need to:

0. set up your cluster with all the perquisites it needs
   * cert-manager-operator
   * cert-manager/ClusterIssuer named local-trusted-issuer
     * example: https://github.com/itewk/lab-rh-itewk/tree/main/ocp/sno-auth-test/kustomize/cert-manager
   * if testing with an external IDP then either have one of those and configure it, or deploy Keycloak and configure it as an OIDC provider for OpenShift
     * example: https://github.com/itewk/lab-rh-itewk/tree/main/ocp/sno-auth-test/kustomize/keycloak
     * of course if deploying keycloak you will need persistent storage for the postgresql db
       * example: https://github.com/itewk/lab-rh-itewk/tree/main/ocp/sno-auth-test/kustomize/lvms-operator
   * the kubeconfig for a separate JWT authentication webhook
     * example: https://github.com/itewk/authn-webhook

0. update all the URLs in [deploy](./deploy/) from `sno-auth-testing.lab.rh-itewk.internal` to whatever your domain is

0. update [authn-webhook.kubeconfig](./deploy/files/authn-webhook.kubeconfig) files `clusters[0].cluster.certificate-authority-data` parameter with the value from the `Secret/multiplexer-proxy-webhook/webhook-route-tls` generated by the `Certificate/multiplexer-proxy-webhook/webhook-route`. This of course means you have to deploy the resources to get the Certificate resource in place and then update the kubeconfig file. So you can see why the effort has not (yet?) been put in to fully automate this
    ```sh
    oc get secret webhook-route-tls -n multiplexer-proxy-webhook -o jsonpath='{.data.ca\.crt}'
    ```

0. update [openshift-oauth-webhook.kubeconfig](./deploy/files/openshift-oauth-webhook.kubeconfig) with the contents of `Secrete/openshift-config/webhook-authentication-integrated-oauth`.

   WARNING: Be sure not to commit this file as it contains a private key. Again another reason this hasn't been fully automated as then would need to deal with a Vault of some sort and ESO.

   ```sh
   oc get secret webhook-authentication-integrated-oauth -n openshift-config -o yaml | yq eval 'del(.metadata.managedFields, .metadata.uid, .metadata.resourceVersion, .metadata.creationTimestamp, .metadata.generation, .status)'
   ```

0. probably deal with a bunch of other hard coded stuff I didn't think about

### Hot Deployment
Again, it can not be stated enough that you do the Testing procedure before doing this. Or you know, YOLO it if you want.

Assuming all the testing looked good then in theory putting this authentication in place is as simple as following the documentation for install a webhook.

0. an example of steps 1 and 2 can be fond here: https://github.com/itewk/lab-rh-itewk/tree/main/ocp/sno-auth-test/kustomize/cluster/authentication
0. create a secret in openshift-config with a kubeConfig value with the KubeConfig to the multiplexer-proxy-webhook
    ```sh
    # example
    kind: Secret
    apiVersion: v1
    metadata:
      name: multiplexer-proxy-webhook
      namespace: openshift-config
    stringData:
      kubeConfig: |
        apiVersion: v1
        kind: Config
        clusters:
        - name: multiplexer-proxy-webhook
            cluster:
            server: https://multiplexer-proxy-webhook.apps.TODO_OCP_DOMAIN
            certificate-authority-data: # TODO your multiplexer-proxy-webhook route CA
            tls-server-name: multiplexer-proxy-webhook.apps.TODO_OCP_DOMAIN
        users:
        - name: multiplexer-proxy-webhook
            user:
            token: "unused"
        contexts:
        - name: multiplexer-proxy-webhook
            context:
            cluster: multiplexer-proxy-webhook
            user: multiplexer-proxy-webhook
        current-context: multiplexer-proxy-webhook
    type: Opaque
   ```
0. Update OpenShift Authentication resource to point at the secret
    ```sh
    apiVersion: config.openshift.io/v1
    kind: Authentication
    metadata:
      name: cluster
    spec:
      type: None
      webhookTokenAuthenticator:
        kubeConfig:
        name: multiplexer-proxy-webhook
    ```
0. Wait with baited breath for the node roll outs to happen
0. Test

    0. try logging into the console using whatever IDPs you have configured
    0. try minting a token with your JWT webhook minter and using that token to log in via the CLI

## Testing
Before updating the OpenShift authentication to actually use this webhook it is **STRONGLY** suggested you test it first as it is not very fun to recover a cluster who's authentication is completely borked. Ask me how I know.

Assuming you got it all deployed and have a running multiplexer-proxy-webhook with clean logs then you can try the following.

### Test authenticating to OpenShift Oauth

attempt to authenticate with the multiplexer-proxy-webhook using a token from openshift oauth
```sh
export OCP_OAUTH_TOKEN=$(oc whoami -t)
export MULTIPLEXER_PROXY_WEBHOOK=$(oc get route webhook -n multiplexer-proxy-webhook --template='{{ .spec.host }}' )
curl -X POST https://${MULTIPLEXER_PROXY_WEBHOOK} \
  -k \
  -H "Content-Type: application/json" \
  -d '{
    "kind": "TokenReview",
    "apiVersion": "authentication.k8s.io/v1",
    "metadata": {},
    "spec": {
        "token": "${OCP_OAUTH_TOKEN}",
        "audiences": [
            "https://kubernetes.default.svc"
        ]
    },
    "status": {
        "user": {}
    }
}'
```

expected output
```sh
{
    "kind": "TokenReview",
    "apiVersion": "authentication.k8s.io/v1",
    "metadata": {
        "creationTimestamp": null,
        "managedFields": [
            {
                "manager": "Go-http-client",
                "operation": "Update",
                "apiVersion": "authentication.k8s.io/v1",
                "time": "2026-04-17T02:48:23Z",
                "fieldsType": "FieldsV1",
                "fieldsV1": {
                    "f:spec": {
                        "f:token": {}
                    }
                }
            }
        ]
    },
    "spec": {
        "token": "sha256~REDACTED"
    },
    "status": {
        "authenticated": true,
        "user": {
            "username": "kube:admin",
            "groups": [
                "system:cluster-admins"
            ],
            "extra": {
                "scopes.authorization.openshift.io": [
                    "user:full"
                ]
            }
        },
        "audiences": [
            "https://kubernetes.default.svc"
        ]
    }
}
```

expected multiplexer-proxy-webhook pod logs
```sh

2026/04/17 04:48:34 INFO Writing response from upstream webhook back to requestor requestor=10.128.0.2:53016 webhook=https://172.30.204.56:443/apis/oauth.openshift.io/v1/tokenreviews
```

### Test authenticating to JWT webhook

attempt to authenticate with the multiplexer-proxy-webhook using a token from JWT webhook
```sh
export JWT_WEBHOOK_TOKEN= # token minuted from the private key of your JWT webhook, such as by using the mint command of https://github.com/itewk/authn-webhook
export MULTIPLEXER_PROXY_WEBHOOK=$(oc get route webhook -n multiplexer-proxy-webhook --template='{{ .spec.host }}' )
curl -X POST https://${MULTIPLEXER_PROXY_WEBHOOK} \
  -k \
  -H "Content-Type: application/json" \
  -d '{
    "kind": "TokenReview",
    "apiVersion": "authentication.k8s.io/v1",
    "metadata": {},
    "spec": {
        "token": "${JWT_WEBHOOK_TOKEN}",
        "audiences": [
            "https://kubernetes.default.svc"
        ]
    },
    "status": {
        "user": {}
    }
}'
```

expected output
```sh
{
    "apiVersion": "authentication.k8s.io/v1",
    "kind": "TokenReview",
    "spec": {
        "token": ""
    },
    "status": {
        "authenticated": true,
        "user": {
            "username": "test-06",
            "uid": ""
        }
    }
}
```

expected multiplexer-proxy-webhook pod logs
```sh
2026/04/17 04:57:59 INFO Forwarding request to upstream webhook requestor=10.128.0.2:45376 webhook=https://authn-webhook.apps.sno-auth-testing.lab.rh-itewk.internal
2026/04/17 04:57:59 INFO Writing response from upstream webhook back to requestor requestor=10.128.0.2:45376 webhook=https://authn-webhook.apps.sno-auth-testing.lab.rh-itewk.internal
```

if using (authn-webhook)[https://github.com/itewk/authn-webhook] example then those logs should look like
```sh
2026/04/17 04:57:59 INFO Review Response authenticated=true user="&{Username:test-06 UID: Groups:[] Extra:map[]}" error=""
```
